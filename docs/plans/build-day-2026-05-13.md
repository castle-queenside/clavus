## Critical Path Hardening (1.5 hrs)

### A. `pull_snapshot_blobs` — Noisy Failures

Currently: if 1 of 20 blobs fails to write, the function `pass`es silently. User sees "3 blobs" but 1 was skipped.

**Fix:** Track failures, report at end:
```
pull_snapshot_blobs() → (success_count, failed_hashes[])
```
Display: `4/5 audio blobs restored (1 failed — check disk space)`

### B. Relay HEAD Race Condition

When `push_to_remote` updates relay HEAD, there's a window where a concurrent pull sees stale data. Already partially addressed with `last_remote_head`, but the update is non-atomic.

**Investigate:** Is there a window where `push_to_remote` could overwrite a newer relay HEAD? If so, document the risk and add a `force=True` guard for the HEAD write.

### C. Snapshot Without .als — Silent No-op

If you `S` snapshot but the `.als` file was deleted/moved, `save_snapshot` silently saves a snapshot with `als_hash=None`. The user sees "Snapshot saved" but has no file to restore.

**Fix:** If `als_hash` is `None`, add a warning: `⚠️ No .als file found — this snapshot cannot restore`
